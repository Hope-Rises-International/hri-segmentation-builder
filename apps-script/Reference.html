1. Waterfall Hierarchy

Each donor lands in exactly one position. Position priority is top-down — the first match wins.

Toggles fall into two semantic buckets:

GROUP toggles (cohort identity — OFF removes those donors entirely):

  • Cornerstone — OFF removes Cornerstone-flagged donors
  • Sustainer (Miracle Partner) — OFF removes Miracle Partners
  • Major Gift Portfolio — OFF removes donors with Staff Manager
  • Mid-Level — OFF removes the Mid-Level cohort (24-mo cumulative ≥ $750)

RFM toggles (lifecycle position — OFF skips routing, donor falls through):

  • Active Housefile — OFF skips AH assignment; R1+R2 donors stay in universe and may route to other ON positions
  • Lapsed — OFF skips LR assignment
  • Deep Lapsed — OFF skips DL assignment

GROUP OFF removes; RFM OFF skips. This means a Mid-Level donor who is also R1 by RFM will go to ML01 (Mid-Level routing wins at waterfall position 3) regardless of whether Active Housefile is ON or OFF. Cohort identity preserved.

Tier 1.5 New Donor pre-emption (v3.3): donors with Lifecycle_Stage__c == 'New Donor' are removed above the waterfall — first-match-wins doesn't get to override the welcome stream. The welcome series itself runs as a separate workflow with all GROUP toggles OFF.

Eliminated in v3.3: Mid-Level Prospect cohort (sub-$750 active donors now route to AH/LR naturally). New Donor as a waterfall position (promoted to Tier 1.5 above).

Nuclear mode (separate top-level toggle, double-confirmed) forces all GROUP toggles ON and bypasses Tier 2 + recent-gift + freq-cap + holdout — but preserves cohort routing and Tier 1 hard suppressions. See SPEC §6.1 for full bypass/preserve table.

6. Ask String Math

Basis selection per segment family:

  • HPC (npo02__LargestAmount__c) — Active Housefile (AH*), Mid-Level (ML*), Cornerstone (CS01), Major Gift (MJ01), Sustainer (SU01).
  • MRC (npo02__LastOppAmount__c) — Lapsed Recent (LR*), Deep Lapsed (DL*), CBNC (CB01).
  • First Gift — New Donor (ND01) (deprecated path; New Donor is pre-empted at Tier 1.5 in v3.3 and the welcome series uses its own workflow — code retained for future reuse).

v3.3: Mid-Level Prospect (MP01) eliminated; sub-$750 active donors route to AH/LR. Code retained as deprecated.

Ladder formula: AskAmount1 = basis × 1.0, AskAmount2 = basis × 1.5, AskAmount3 = basis × 2.0.

Rounding: UP to nearest $5 below $100, UP to nearest $25 at/above $100. Never round down.

Floor / Ceiling: Floor = $15. Ceiling = $4,975 (rounded down to a $25 increment).

Floor-collapse fallback (2026-04-27): If any raw tier (basis × multiplier) is below the $15 floor, replace the WHOLE ladder with $15 / $25 / $35 as a unit. Never re-floor tiers independently — that produced non-monotonic ladders like 15 / 20 / 15. Fallback ladder values ($15 / $25 / $35) live in src/config.py constants. Editing requires a redeploy.

Mid-Level / Major: ML*, MJ*, MP*, CS01, SU01 segments must always populate AskAmount1/2/3. The lettershop template controls whether the asks render on the printed reply device — builder always supplies the values.

AskAmountLabel: ALWAYS BLANK in the CSV. The "Best Gift of $___" line is a donor fill-in; the lettershop renders the static label and HRI does not anchor it.

7. Scanline & Check Digit

Format: <DonorID:9> <CampaignAppealCode:9> <CheckDigit:1> — 21 chars total, two literal spaces.

DonorID: 9 chars; numeric Constituent_Id zero-padded; S-prefixed pass through unchanged.

CampaignAppealCode: <TypePrefix:1><FY:2><Campaign:2><SegmentCode:4> — e.g. A2651AH01.

Check digit conversion table (alpha → digit; numerics keep value):

A=1B=2C=3D=4E=5F=6G=7H=8I=9
J=1K=2L=3M=4N=5O=6P=7Q=8R=9
S=2T=3U=4V=5W=6X=7Y=8Z=9

7-step algorithm on the 18-char (DonorID + AppealCode) string:

  1. Treat as 18 individual characters.
  2. Replace alpha → digit per table above; numerics keep their value.
  3. Assign alternating weights 1, 2, 1, 2, … across the 18 positions.
  4. Multiply each value × its weight.
  5. If a product > 9, subtract 9; otherwise keep.
  6. Sum all 18 step-5 values.
  7. CheckDigit = (10 − (sum mod 10)) mod 10.

Worked example: 070122327W16B1AJ30 → check digit 6. Full scanline: 070122327 W16B1AJ30 6.

10. Holdout & Floor-Collapse Edge Cases

Per-segment holdout (v3.4): each segment row in the Step 3 scenario editor has its own Holdout % column (default 5; integer range 0–5). The selected fraction of each segment's mailable cohort is suppressed from the Print file and retained in the Matchback with Holdout=TRUE for ROI measurement. The 5% upper bound preserves the measurement-infrastructure cap; 0 means no holdout for that segment. Soft warning (orange) appears below 3% to flag reduced measurement power. Random sampling is seeded per segment so re-runs of the same scenario produce identical Holdout=TRUE row sets.

Floor-collapse fallback: When any tier of the basis-driven ladder falls below the $15 floor, replace the whole ladder with $15 / $25 / $35 as a unit. Never re-floor tiers independently. Avoids non-monotonic outputs like 15 / 20 / 15 for tiny basis values.

Toggle bucket model (v3.3): GROUP toggles (Cornerstone, Sustainer, Major Gift, Mid-Level) remove matching donors from the universe entirely when OFF. RFM toggles (Active Housefile, Lapsed, Deep Lapsed) skip the assignment gate when OFF — the donor stays in the universe and may route to other ON positions. A Mid-Level + R1 donor lands in ML01 even with Active Housefile OFF; cohort identity is preserved. Cornerstone OFF means zero Cornerstone-flagged donors anywhere in the output. New Donor (90-day welcome window) is pre-empted at Tier 1.5 above the waterfall and is not part of the GROUP/RFM toggle set; the welcome series itself runs as a separate workflow.

14. Matchback Row Scope

Matchback's purpose is gift attribution: when a donor responds, the row recovers segment / package / test-cell detail. Rows that don't contribute to that join don't belong in the file.

Included:

  • Mailed donors — Holdout=FALSE AND ExclusionReason="". Match Print 1:1.
  • Holdouts — Holdout=TRUE. Kept regardless of any exclusion flag, for ROI measurement.

Excluded (these rows live in the Suppression Audit Log file in Drive, not Matchback):

  • Holdout=FALSE AND ExclusionReason="quantity_reduction" — Pass 2 budget trim.
  • Holdout=FALSE AND ExclusionReason="missing_constituent_id" — data quality.
  • Holdout=FALSE AND ExclusionReason="duplicate_constituent_id" — data quality.
